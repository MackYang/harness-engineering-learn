## 一、Harness Engineering: Leveraging Codex in an Agent-First World（2025-08）

**核心主题**：零手动编写代码的软件开发实验——通过 Harness 工程实现百万行代码规模的完全 AI 生成

### 1.1 实验概述

**五年成果**：
- 100万行代码完全由 Codex 生成
- 3名工程师驱动，每天平均3.5个PR
- 产品有内部用户和外部alpha测试者
- 预计速度提升10倍

**核心哲学**：
> "Humans steer. Agents execute."
> 人类设定方向，代理执行任务

### 1.2 开发环境设计

**零手动代码原则**：
- 从空git仓库开始
- 初始脚手架（CI配置、包管理、框架）由Codex生成
- AGENTS.md文件本身也由Codex编写
- 每行代码（应用逻辑、测试、CI配置、文档、可观测性）都由AI生成

**代码生成范围**：
- 产品代码和测试
- CI配置和发布工具
- 内部开发者工具
- 文档和设计历史
- 评估框架
- 审查评论和响应
- 管理仓库本身的脚本
- 生产仪表板定义文件

### 1.3 工程师角色重新定义

**从编码到系统设计**：
- 早期进展缓慢不是Codex能力问题，而是环境规范不足
- 工程师主要工作是创建代理能用的工具、抽象和内部结构
- 深度优先工作：将大目标分解为小块（设计、代码、审查、测试）
- 当失败时，识别缺少的能力，使其对代理可理解和可强制执行

**交互方式转变**：
- 工程师通过Prompt与系统交互：描述任务，运行代理，允许打开PR
- Codex使用标准开发工具直接：收集上下文，响应反馈，推送更新
- 人类可以审查PR，但不是必需的，大部分审查工作代理间完成

### 1.4 提高应用可读性

**瓶颈从编码转向QA**：
- 代理吞吐量增加后，瓶颈变为人工QA能力
- 通过使应用UI、日志、指标直接对Codex可读来增强代理能力

**具体实践**：
- **可启动工作树**：应用可按git worktree启动，每个变更一个实例
- **Chrome DevTools集成**：将Chrome DevTools Protocol连接到代理运行时
- **DOM快照和截图**：创建处理DOM快照、截图和导航的技能
- **独立可观测性栈**：每个工作树都有独立的可观测性栈
- **LogQL和PromQL**：代理可查询日志和指标

**示例Prompt**：
- "确保服务启动在800毫秒内完成"
- "这四个关键用户旅程中没有跨度超过两秒"

### 1.5 知识管理系统设计

**从"一本大说明书"到"目录"**：
- **失败尝试**：单一AGENTS.md文件作为百科全书
  - 上下文稀缺资源，大文件挤占任务空间
  - 过多指导等于无指导
  - 瞬间过时，变成规则墓地
  - 难以验证，无法机械检查

**成功实践**：
- **结构化文档目录**：docs/目录作为系统记录
- **短AGENTS.md**（约100行）：作为地图，指向更深层次的真相
- **渐进式披露**：代理从小稳定的入口点开始，学会在哪里寻找

**文档类型**：
- **设计文档**：包含验证状态和核心信念
- **架构文档**：领域和包分层的顶层地图
- **质量文档**：为每个产品领域和架构层分级，跟踪差距
- **计划文档**：轻量级计划用于小变更，执行计划用于复杂工作

### 1.6 代理可读性目标

**优化代理视角**：
- 代码库完全代理生成，首先为Codex可读性优化
- 类似为新工程人员改进代码可导航性
- 代理需要直接从仓库中推理整个业务领域

**内部化原则**：
- 知识必须存储在仓库中，版本化
- 支持组合性、API稳定性、训练集表示的技术更容易被代理建模
- 有时让代理重新实现功能子集比处理公共库的不透明上游行为更便宜

**约束原则**：
- 推向更多可被代理检查、验证和修改的系统形式
- 不仅适用于Codex，也适用于其他代理（如Aardvark）

### 1.7 架构和品味强制执行

**不变量而非微观管理**：
- 文档 alone 无法保持完全代理生成的代码库连贯性
- 通过强制执行不变量，让代理快速发布而不损害基础

**严格分层架构**：
- 每个业务领域分为固定层集
- 严格验证依赖方向和有限的可允许边缘
- 通过自定义linters和结构测试强制执行

**依赖规则**：
```
App Settings领域：
Types → Config → Repo → Service → Runtime → UI
交叉关注点（auth、connectors、telemetry、feature flags）通过单个接口：Providers
```

**品味不变量**：
- 静态强制执行结构化日志
- 架构和类型命名约定
- 文件大小限制
- 平台特定的可靠性要求

### 1.8 合并哲学转变

**高吞吐环境下的工程规范**：
- 仓库运行最小阻塞合并门
- PR短暂存在
- 测试失败通常通过后续运行解决，而非无限期阻塞
- 在代理吞吐量远超人类注意力的系统中，纠错便宜，等待昂贵

### 1.9 自主性层次

**端到端驱动功能**：
- 测试、验证、审查、反馈处理、恢复编码到系统后，仓库跨越重要门槛
- 单个Prompt下，代理可以：
  1. 验证代码库当前状态
  2. 复制报告的bug
  3. 录制失败视频
  4. 实现修复
  5. 通过驱动应用验证修复
  6. 录制解决视频
  7. 打开PR
  8. 响应代理和人类反馈
  9. 检测和修复构建失败
  10. 仅在需要判断时升级到人类
  11. 合并变更

### 1.10 熵和垃圾收集

**模式复制与漂移**：
- Codex复制已存在的模式，甚至是不均匀或次优的
- 随时间推移，这必然导致漂移

**"黄金原则"**：
- 将"黄金原则"编码到仓库中
- 构建定期清理流程
- 意见化的机械规则保持代码库可读和一致

**示例原则**：
1. 优先共享实用包而非手写助手，保持不变量集中
2. 不进行"YOLO风格"数据探测，验证边界或依赖类型化SDK

**垃圾收集功能**：
- 技术债务是高息贷款：连续小额偿还比痛苦爆发更好
- 人类品味一次捕获，然后在每行代码上持续强制执行
- 每天捕获和解决坏模式，防止它们在代码库中传播数天或数周

### 1.11 仍在学习的内容

**长期挑战**：
- 完全代理生成系统中架构连贯性如何随时间演变
- 人类判断在何处增加最大杠杆，如何编码该判断
- 随着模型能力继续提高，系统如何发展

**核心洞察**：
- 构建软件仍然需要纪律，但纪律显示在脚手架而非代码中
- 保持代码库连贯的工具、抽象和反馈循环越来越重要

### 1.12 对 Harness Engineering 的启示

**新的纪律形式**：
1. **脚手架纪律**：专注于环境设计、反馈循环和控制系统的工程
2. **约束即加速**：严格边界允许速度而不衰变或架构漂移
3. **模式提取**：将常见失败模式转化为永久约束
4. **代理视角设计**：从代理的可读性角度优化一切

**人类价值的转移**：
- 从编写代码到设计系统
- 从微观实现到宏观架构
- 从即时反馈到长期可维护性

## 二、知识体系更新总结

### 新增核心概念

| 概念 | 来源 | 一句话描述 |
|------|------|-----------|
| **零手动代码原则** | OpenAI Harness Engineering | 从开始到结束，所有代码都由AI生成，人类只做方向引导 |
| **代理可读性目标** | OpenAI Harness Engineering | 代码库完全优化代理可读性，使其能直接推理业务领域 |
| **严格分层架构** | OpenAI Harness Engineering | 固定层集和严格依赖方向，通过机械强制执行保持连贯性 |
| **端到端自主性** | OpenAI Harness Engineering | 单个Prompt下代理可完成从bug发现到PR合并的完整流程 |
| **垃圾收集系统** | OpenAI Harness Engineering | "黄金原则"和定期清理，持续技术债务管理 |

### 与其他来源的交叉验证

1. **与Loop Engineering的关系**：
   - OpenAI的实践体现了Loop Engineering的五大构件：Automations、Worktrees、Skills、Plugins、Sub-agents+State
   - "Humans steer. Agents execute"正是Loop Engineering的核心哲学

2. **与Anthropic实践的呼应**：
   - 强调代理可读性与Anthropic的"代理只能直接操作上下文中的内容"一致
   - 工具设计理念相似：严格限制、清晰边界

3. **与Harness Engineering的共同点**：
   - 都将失败转化为永久约束
   - 都强调脚手架重于代码
   - 都重视自动化测试和验证

## 三、实践指导

### 架构设计原则
1. **严格边界优先**：在系统早期就建立严格的分层架构
2. **代理视角思考**：每个设计决策都要考虑"代理能理解这个吗？"
3. **渐进式披露**：从简单稳定的入口点开始，逐步揭示复杂性

### 工程实践原则
1. **失败即信号**：每个代理错误都是永久约束的信号
2. **无手工代码**：建立让AI能独立完成所有工作的环境
3. **自动化验证**：将测试、类型检查、lint等融入工作流

### 团队角色转变
1. **从编码到设计**：工程师转向环境、反馈循环和控制系统的设计
2. **审查自动化**：大部分审查工作由代理间完成
3. **知识管理**：建立结构化的知识系统，代理能自主查找和学习
