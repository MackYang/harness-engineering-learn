# Agent 工作流模式

> 来源：Anthropic — [Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents)
> 核心思想：最成功的实现不使用复杂框架，而是用简单、可组合的模式

## 基础构建块：增强型 LLM

所有 Agent 系统的基础是增强了检索、工具和记忆的 LLM。模型主动生成搜索查询、选择工具、决定保留什么信息。

## 五种工作流模式

### 1. Prompt Chaining（提示链）

将任务分解为顺序步骤，每个 LLM 调用处理上一个的输出。可在中间步骤添加编程检查（"gate"）确保流程正常。

**适用场景：**
- 任务可以干净地分解为固定子任务
- 用延迟换取更高准确度
- 示例：先生成营销文案，再翻译

**关键：** 每一步都变成更简单的任务，准确度更高。

### 2. Routing（路由）

对输入分类并导向专门的后续任务。关注点分离，构建更专门的 prompt。

**适用场景：**
- 复杂任务有明显的类别需要分别处理
- 简单问题用小模型（成本低），复杂问题用大模型
- 示例：客户服务查询分流

### 3. Parallelization（并行化）

LLM 同时处理任务并聚合输出。两种变体：
- **Sectioning**：任务拆分为独立子任务并行执行
- **Voting**：同一任务运行多次获得多样输出

**适用场景：**
- 子任务可以并行化提速
- 需要多角度/多次尝试提高置信度
- 复杂任务中每个关注点独立处理更好

### 4. Orchestrator-Workers（编排者-工作者）

一个中央 LLM（编排者）动态分解任务，委派给工作者 LLM，综合结果。

**适用场景：**
- 任务无法预测所需子任务数量和类型
- 需要灵活的分解策略
- 示例：大规模代码变更，每个文件需要不同类型的修改

### 5. Evaluator-Optimizer（评估者-优化者）

一个 LLM 生成输出，另一个 LLM 提供评估和反馈，循环直到满足质量标准。

**适用场景：**
- 有明确的评估标准
- 迭代改进有价值
- 示例：代码审查循环

## 自主 Agent

当工作流不够灵活时，Agent 是更好的选择：
- LLM 动态决定自己的流程和工具使用
- 维持对如何完成任务的自主控制
- 适用于需要灵活性和规模化的场景

## 设计原则

1. **从最简单的方案开始** — 不要上来就构建 Agent 系统
2. **理解底层代码** — 用框架时要理解原理
3. **可组合性 > 复杂性** — 简单可组合的模式胜过复杂框架
4. **延迟和成本的权衡** — Agent 系统通常用更多资源换更好效果

## 与 Harness Engineering 的关联

这些工作流模式是构建 Harness 的基础：
- **Prompt Chaining** → 增量式开发（一次一个 feature）
- **Evaluator-Optimizer** → Generator ≠ Evaluator 原则
- **Orchestrator-Workers** → 任务卡驱动执行（编排者分配任务卡）
- **Routing** → 按优先级路由任务卡
