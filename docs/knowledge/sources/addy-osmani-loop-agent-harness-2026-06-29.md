## 一、Loop Engineering（循环工程）

**核心主题**：从直接提示代理到设计提示代理的系统，五大构件构建自动化循环

### 1.1 核心哲学转变

**旧模式 vs 新模式**：
- **旧模式**：你直接提示代理，读取响应，再写下一个提示
- **新模式**：你构建一个小系统，发现工作、分配工作、检查工作、记录完成状态，决定下一步

**关键观点**：
> "You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents."
> "我不提示Claude了。我运行提示Claude的循环，并决定做什么。我的工作是编写循环。"

### 1.2 五大构件（The Five Pieces）

#### 1. Automations（自动化）
**功能**：按计划运行，自动发现和分类任务
- **Codex**：Automations标签页，选择项目、提示、频率、环境
- **Claude Code**：`/loop`、`/goal`、cron任务、钩子、GitHub Actions

**关键特性**：
- 运行结果存入Triage收件箱
- 可以调用技能保持可维护性
- 解决了"负担过轻"问题：人类不再需要四处检查

#### 2. Worktrees（工作树）
**功能**：隔离并行工作，避免文件冲突
- **Codex**：内置的线程级工作树
- **Claude Code**：`--worktree`标志、子agent的隔离设置

**重要性**：
- 解决了多个代理同时工作的碰撞问题
- Git工作树提供了真正的并行隔离

#### 3. Skills（技能）
**功能**：将项目知识编码化，避免重复解释
- **格式**：包含SKILL.md文件夹和可选的脚本、资源
- **调用**：`$name`或隐式匹配

**意图债务**解决：
- 代理每次会话都从冷启动开始
- 技能将意图写在外面，成为代理每次运行都读取的规范

#### 4. Plugins/Connectors（插件/连接器）
**功能**：将代理连接到你现有的工具
- **基础**：MCP（Model Context Protocol）
- **分发**：插件将连接器和技能打包在一起

**价值**：
- 从"这是修复"到"打开PR、更新Linear、CI通过后发通知"的飞跃

#### 5. Sub-agents（子代理）
**功能**：让一个代理产生想法，另一个检查它
- **分离**：编写者与检查者分离
- **差异**：有时使用不同模型，不同推理强度

**必要性**：
- 编写代码的模型在给自己的作业打分时过于宽容
- 验证代理真正可信，代理运行时无人监督

#### 6. State（状态）
**功能**：记住已完成和接下来要做的事
- **存储**：Markdown文件、Linear看板
- **位置**：单次对话之外的地方

**重要性**：
- 模型在运行之间会忘记一切
- 状态必须存储在磁盘上，而不是在上下文中

### 1.3 实际循环示例

**每日CI修复循环**：
1. **自动化**：早上在仓库上运行，调用故障分类技能
2. **输入**：读取昨日CI失败、开放问题、最近提交
3. **工作树**：为值得的修复打开隔离的工作树
4. **子代理**：
   - 草案代理编写修复
   - 审查代理对照项目技能和现有测试审查草案
5. **连接器**：循环打开PR并更新工单
6. **状态**：记录尝试、通过、仍开放的内容

### 1.4 循环的局限性

**三个仍然存在的问题**：
1. **验证仍然依赖你**：无人监督的循环也会无监督地犯错
2. **理解仍然退化**：循环越快地发布你未编写的代码，存在的与你实际获得的差距越大
3. **舒适姿势的危险**：循环自行运行时，很容易停止有意见，接受它给出的任何结果

**关键观点**：
> "Build the loop. But build it like someone who intends to stay the engineer, not just the person who presses go."
> "构建循环。但要像打算继续担任工程师的人那样构建，而不是仅仅按下按钮的人。"

## 二、Agent Harness Engineering（代理工程）

**核心主题**：将代理的支架视为真实工件，每当代理滑落时收紧

### 2.1 核心定义

**基础方程**：
> Agent = Model + Harness. If you're not the model, you're the harness.
> 代理 = 模型 + Harness。如果你不是模型，你就是Harness。

**Harness包含**：
- 系统提示、CLAUDE.md、AGENTS.md、技能文件、子代理提示
- 工具、技能、MCP服务器及其描述
- 捆绑的基础设施（文件系统、沙箱、浏览器）
- 编排逻辑（子代理生成、交接、模型路由）
- 钩子和中间件（压缩、继续、lint检查）
- 可观测性（日志、追踪、成本和延迟计量）

### 2.2 "技能问题"的重构

**传统思维**：
- 代理做蠢事 → 归咎于模型 → 等待下一版本

**Harness工程思维**：
- 代理做蠢事 → 信号 → 将失败转化为永久约束

**具体案例**：
- 代理不了解约定 → 添加到AGENTS.md
- 代理运行破坏性命令 → 添加阻止它的钩子
- 代理在40步任务中迷失 → 分解为规划者和执行者
- 代理持续"完成"损坏的代码 → 将类型检查回压信号接入循环

### 2.3 拉吉特效应（The Ratchet）

**核心原则**：将代理的每个错误视为永久信号

**示例**：
- 代理发布带有注释测试的PR → 添加新规则"永不注释测试；删除或修复它们"
- 添加pre-commit钩子，在差异中grep .skip(和xit(
- 添加审查子代理，将注释测试标记为阻塞项

**要求**：
- 只有在看到实际失败时才添加约束
- 只有当有能力模型使其冗余时才移除约束
- 好的AGENTS.md中的每一行都应该追溯到具体的失败

### 2.4 从行为反向推导

**设计模式**：从你想要的行为推导出交付它的Harness组件

#### 文件系统和Git：持久状态
- 代理的原始操作空间
- 数据、代码、文档的读取位置
- 中间工作的卸载位置
- 多个代理和人类通过共享文件协调的表面

#### Bash和代码执行：通用工具
- ReAct循环：推理 → 通过工具调用采取行动 → 观察结果 → 重复
- bash让代理即时构建所需工具
- 从单一厨房工具到整个厨房的转变

#### 沙箱和默认工具
- 代码运行的安全隔离环境
- 预安装的语言运行时和包
- Git和测试CLI
- 无头浏览器用于网络交互

#### 内存和搜索：持续学习
- 模型没有训练之外的知识
- AGENTS.md等内存文件标准
- Web搜索和MCP工具（如Context7）

#### 上下文衰减
**三种技术**：
1. **压缩**：接近窗口满时，智能总结和卸载旧上下文
2. **工具调用卸载**：将大输出卸载到文件系统
3. **渐进式披露技能**：只在任务需要时显示工具和MCP

#### 长期执行：拉吉环（Ralph Loops）
**核心概念**：
- 钩子拦截模型的退出尝试，将原始提示重新注入新鲜上下文
- 每次迭代都重新开始，但通过文件系统读取前一状态
- 将单会话代理转换为多会话代理的简单技巧

**规划/生成/评估分离**：
- 生成与评估分离为独立代理优于自我评估
- "冲刺合同"：生成者和评估者事先协商"完成"的含义
- 事前写下完成条件比任何提示变更都更能捕获范围漂移

#### 钩子：执行层
**原则**：成功是沉默的，失败是冗长的
- 在特定生命周期点运行的脚本
- 阻止破坏性bash命令
- 在写入时自动格式化

### 2.5 AGENTS.md和工具选择

**最佳实践**：
1. **保持简短**：HumanLayer保持在60行以下
2. **每行都有价值**：规则应追溯到具体的过去失败或硬性外部约束
3. **工具选择**：10个专注的工具优于50个重叠的工具
4. **安全考虑**：MCP服务器是代理读取的可信文本

### 2.6 生产环境中的Harness

**Claude Code架构**：
- 知识层：上下文注入
- 内存存储：循环状态
- 工作树隔离器
- 权限网关：破坏性操作钩子
- 多代理层：子代理上下文防火墙
- 工具分发注册表：MCP服务器和bash的接入点

### 2.7 Harness的演进

**关键洞察**：
> "As models improve, the space of interesting harness combinations doesn't shrink. It moves."
> "随着模型改进，有趣的Harness组合空间不会缩小。它会移动。"

**模型-Harness训练循环**：
- 今天的代理产品在Harness循环中进行后训练
- 模型在Harness设计者认为它应该擅长的事情上变得更好
- 这就是为什么Opus 4.6在Claude Code中感觉不同的原因

**Harness-as-a-Service（HaaS）**：
- 从在LLM API上构建（给你一个完成）到在Harness API上构建（给你一个运行时）
- Claude Agent SDK、Codex SDK、OpenAI Agents SDK都指向同一方向

### 新增核心概念

| 概念 | 来源文章 | 一句话描述 |

| **拉吉效应** | Agent Harness | 每个错误转化为永久约束，制度性改进 |
| **行为反向推导** | Agent Harness | 从期望行为推导Harness组件 |

| **持续学习机制** | Agent Harness | AGENTS.md等内存文件实现跨会话学习 |

| **HaaS框架** | Agent Harness | Harness-as-a-Service的新模式 |

### 跨文章交叉洞察

1. **从提示到循环的转变**：
   - Loop Engineering明确提出从直接提示到设计循环
   - Agent Harness Engineering提供了实现循环的架构基础
   - 两者都是提高代理能力层次的方法

2. **错误即信号**：
   - Loop Engineering中的意图债务通过技能解决
   - Agent Harness Engineering中的拉吉效应将错误转化为永久约束
   - 两者都强调将失败视为改进机会

3. **组件解耦**：
   - Loop Engineering的五大构件相互独立
   - Agent Harness Engineering强调各组件的特定职责
   - 两者都支持模块化和可组合性

4. **代理能力分层**：
   - 底层：模型本身
   - 中层：Harness组件（技能、工具、钩子等）
   - 上层：Loop架构（自动化、工作树、子代理等）

### 实践指导

#### 循环设计原则
1. **从简单开始**：先实现基本循环，再添加复杂功能
2. **明确职责分离**：让不同的代理处理不同阶段
3. **状态持久化**：确保循环能记住进度
4. **安全边界**：使用沙箱和钩子保护系统

#### Harness设计原则
1. **从行为推导**：先定义想要的行为，再设计组件
2. **最小化约束**：只在失败时添加约束
3. **渐进式披露**：只显示当前任务需要的工具和文档
4. **可观测性**：添加日志、追踪和成本监控

#### 技能编写原则
1. **简洁**：保持AGENTS.md简短，每行都有价值
2. **具体**：每个规则都应该追溯到具体失败
3. **结构化**：使用SKILL.md组织项目知识
4. **渐进式加载**：按需加载技能内容

## 四、与其他来源的对比

### 与OpenAI Harness Engineering的关系
- **互补性**：OpenAI的零手动代码原则可以通过Addy的循环架构实现
- **扩展性**：五大构件扩展了OpenAI的脚手架纪律概念
- **实践性**：Addy的具体构件为OpenAI的理论提供了实现路径

### 与Anthropic实践的呼应
- **安全理念**：两者都强调环境防御的重要性
- **长期工作**：上下文衰减和持续学习都是长期工作的关键
- **组件分离**：子代理分离验证的实践在两个来源都有强调

### 整合框架
1. **战略层**：Loop Engineering（大方向、自动化流程）
2. **战术层**：Agent Harness Engineering（组件设计、约束管理）
3. **执行层**：OpenAI/Harness最佳实践（脚手架纪律、零手动代码）
