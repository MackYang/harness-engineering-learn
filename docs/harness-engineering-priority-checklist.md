# Harness-Engineering 主子清单（按优先级）

> 目标：在不覆盖全部公司流程的前提下，建立可持续自我进化的 harness-engineering 体系。
> 核心：把“度量驱动、失败反哺、自动执行”作为最高优先级硬机制。

## P-1：实施前准备（Go/No-Go 门槛）

### P-1-1 试点范围与责任冻结（Blocker）
- [ ] 选定试点项目（范围、周期 4-8 周、业务边界）
- [ ] 冻结 RACI（Owner/Reviewer/Approver/Incident Commander）
- [ ] 冻结初始自治等级（L1/L2）与回退条件

验收标准：
- [ ] 项目章程、RACI、自治等级策略完成签字确认

### P-1-2 工程基础可执行（Blocker）
- [ ] CI 最小能力可用（至少 lint + test 稳定运行）
- [ ] 代码托管权限模型确认（合并权限、豁免权限）
- [ ] 发布与回滚路径完成至少 1 次演练

验收标准：
- [ ] 可证明在 1 次演练内完成“发布 -> 回滚 -> 恢复”

### P-1-3 数据与风险基线（Blocker）
- [ ] 指标数据源打通（Git/CI/Issue）并能自动出周报
- [ ] 高风险目录冻结（权限、计费、生产配置、数据删除）
- [ ] 复盘 SLA（24h/7d）与模板投入实际使用

验收标准：
- [ ] 首周即可自动产出基线指标且有首份复盘记录

### P-1-4 首轮执行包（Blocker）
- [ ] 首批任务卡 owner 与截止日期填充完成
- [ ] 统一证据格式明确（命令输出、验收截图/日志、风险说明）
- [ ] 周节奏日历冻结（周会/月会/季度评审）

验收标准：
- [ ] 首轮 10-12 张任务卡可直接进入执行，无需额外解释会

### P-1-5 强化项（Strongly Recommended）
- [ ] AI 执行环境标准化（模型、上下文来源、命令白名单）
- [ ] Issue/PR 模板字段校验自动化
- [ ] 最小 eval 集覆盖至少 1 条核心业务流程
- [ ] 值班与升级路径演练完成
- [ ] 法务/合规边界完成确认

验收标准：
- [ ] 强化项完成率 >= 80% 后再进入大规模扩圈

## P0（最高优先级，必须先完成）：自我进化三大硬机制

### P0-1 度量驱动（Metrics-Driven）
- [ ] 建立统一指标看板（周维度）
- [ ] 交付指标：DORA（Lead Time、Deploy Frequency、Change Failure Rate、MTTR）
- [ ] 体验指标：SPACE（至少覆盖满意度、协作效率、流效率）
- [ ] 质量指标：Eval 通过率、回归失败率、人工返工率
- [ ] 设置阈值与告警规则（红/黄/绿）

验收标准：
- [ ] 所有指标自动采集，不依赖人工填报
- [ ] 每周例会固定复盘指标趋势与异常

产出物：
- `docs/metrics/engineering-scorecard.md`
- 可视化看板链接（Grafana/Datadog/BI）

### P0-2 失败反哺（Failure-to-Policy Loop）
- [ ] 定义失败事件类型：回滚、线上事故、误改、越权、评测失效
- [ ] 固化复盘 SLA：24h 初判，7 天内完成改进闭环
- [ ] 每次复盘必须输出至少一项可执行改进：
- [ ] 新测试（unit/integration/eval）
- [ ] 新策略（Policy-as-Code）
- [ ] 新规则（代码规范/模板/审批门禁）

验收标准：
- [ ] 复盘项有 owner、有截止日期、有状态追踪
- [ ] 同类问题复发率逐月下降

产出物：
- `docs/incidents/postmortem-template.md`
- `docs/incidents/lessons-learned-log.md`

### P0-3 自动执行（Automation as Enforcement）
- [ ] 将规则全部转为 CI 可执行门禁
- [ ] 引入 Policy-as-Code（如 OPA）处理高风险规则
- [ ] 定义禁止直通场景：未过测试/未过 eval/高风险未审批
- [ ] 建立规则版本管理与审计记录

验收标准：
- [ ] 人工“口头规范”全部落地为可执行检查
- [ ] 关键分支无绕过门禁的合并记录

产出物：
- `.github/workflows/`（或等价 CI 配置）
- `policy/`（策略规则目录）

## P1：代码库真相源与统一入口

### P1-1 真相源文档收敛
- [ ] 建立并维护 `ARCHITECTURE.md`
- [ ] 建立并维护 `CONTRIBUTING.md`
- [ ] 建立 `AGENTS.md`（智能体入口索引）
- [ ] ADR（架构决策记录）入库并可追溯

验收标准：
- [ ] 关键模块均有 owner、边界与依赖说明
- [ ] 新人可在 30 分钟内定位核心信息

### P1-2 统一执行入口（Human + Agent）
- [ ] 建立统一命令：build/test/lint/eval/release
- [ ] 命令输出统一格式，便于机器解析
- [ ] 本地与 CI 执行行为一致

验收标准：
- [ ] 人和 agent 不再绕过统一命令执行路径

产出物：
- `Makefile` 或 `Taskfile.yml`
- `docs/runbooks/dev-workflow.md`

### P1-3 模板化输入输出
- [ ] Issue 模板：背景、边界、验收标准、非目标、风险级别
- [ ] PR 模板：变更摘要、验证证据、风险评估、回滚方案

验收标准：
- [ ] 新增 PR 的模板字段完整率 >= 95%

## P2：质量、安全、供应链门禁

### P2-1 基础质量门禁
- [ ] lint/type-check/unit/integration 全部门禁化
- [ ] 高风险目录（认证、计费、权限）强制更高测试覆盖

验收标准：
- [ ] 主分支始终保持可部署状态

### P2-2 AI/LLM 专项安全
- [ ] 建立提示注入、数据泄露、工具滥用等对抗用例
- [ ] 将 OWASP LLM Top 10 映射为检查清单

验收标准：
- [ ] 每次发布前自动跑关键安全 eval

### P2-3 供应链可信
- [ ] 生成 SBOM
- [ ] 构建产物保留 provenance（来源证明）
- [ ] 按季度推进 SLSA 级别目标

验收标准：
- [ ] 关键构建可追溯到源码、依赖、构建环境

## P3：发布可靠性与风险治理

### P3-1 SLO 与 Error Budget
- [ ] 关键服务定义 SLO
- [ ] 预算耗尽触发变更冻结机制

验收标准：
- [ ] 速度与稳定性冲突时有明确决策规则

### P3-2 渐进发布与自动回滚
- [ ] Canary / Staged rollout
- [ ] 触发阈值自动回滚
- [ ] 发布后观测窗口与复核机制

验收标准：
- [ ] 高风险发布具备可演练回滚路径

### P3-3 AI 风险治理框架
- [ ] 落地 NIST AI RMF：Govern/Map/Measure/Manage
- [ ] 角色职责与审批边界制度化

验收标准：
- [ ] 风险控制要求可审计，可重复执行

## P4：扩圈与组织化运营

### P4-1 Pilot 扩圈机制
- [ ] 连续 2 周达标后才扩到下一个团队
- [ ] 扩圈前完成 SOP 与培训包

验收标准：
- [ ] 扩圈后指标无明显回撤

### P4-2 平台化 Golden Path
- [ ] 建立标准脚手架（默认安全与观测配置）
- [ ] 服务目录包含 owner、依赖、SLO、状态

验收标准：
- [ ] 新服务接入成本显著下降

### P4-3 周期运营
- [ ] 周会：指标、异常、规则更新
- [ ] 月会：复盘模式与流程债
- [ ] 季度：策略有效性与误报率审计

验收标准：
- [ ] 规则数量增长同时误报率可控

## P5：长期成熟度与规模化优化

### P5-1 跨项目资产复用库
- [ ] 建立可复用资产库：策略模板、eval 数据集、任务卡模板、发布模板
- [ ] 建立资产版本管理与变更日志
- [ ] 建立跨项目复用率指标（按季度）

验收标准：
- [ ] 新项目 70% 以上可直接复用已有资产

### P5-2 误报率与规则成本优化
- [ ] 对 CI/Policy/eval 建误报率与耗时监控
- [ ] 建立规则分级（强阻断/弱告警/观察中）
- [ ] 每月进行规则降噪与成本优化

验收标准：
- [ ] 关键门禁误报率持续下降
- [ ] 平均流水线时长在可接受范围内

### P5-3 人机协作组织能力
- [ ] 建立角色矩阵（Owner、Reviewer、Approver、Incident Commander）
- [ ] 建立值班与升级机制（包括 AI 失败升级路径）
- [ ] 建立培训与认证路径（新成员/新团队接入）

验收标准：
- [ ] 团队交接时不依赖单点专家
- [ ] 关键审批链在 SLA 内完成

### P5-4 L3/L4 稳定性验证周期
- [ ] 建立高自治等级专项验证（连续 8-12 周）
- [ ] 记录升级与回退事件，评估触发根因
- [ ] 形成“自治等级健康报告”（季度）

验收标准：
- [ ] 高自治等级下质量与稳定性不劣化
- [ ] 回退触发率可控且可解释

## 全局完成定义（DoD）
- [ ] 同类问题复发率持续下降
- [ ] 人工 review 负载下降且质量不下降
- [ ] 交付速度提升且 Change Failure Rate 不上升
- [ ] 关键流程全部可自动化执行和审计
- [ ] 跨项目复用率逐季提升
- [ ] 关键规则误报率与执行成本逐季优化

## 执行顺序建议
1. 先完成 P-1（Go/No-Go 门槛）
2. 再完成 P0（必须 100%）
3. 然后完成 P1（建立统一工作面）
4. 并行推进 P2 + P3（质量安全与可靠性）
5. 再做 P4（规模化复制）
6. 最后持续执行 P5（长期成熟度优化）
