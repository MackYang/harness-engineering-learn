## 一、How we contain Claude across products（2026-06-23）

**核心主题**：三种代理隔离模式与安全教训 - 从环境层防御角度探讨代理安全

### 1.1 风险分类与防御组件

**三类安全风险**：
1. **用户误用**：用户恶意或粗心导致有害行为
2. **模型行为不当**：代理执行非预期的有害行为
3. **外部攻击者**：通过工具、文件或网络访问攻击代理

**三个防御组件**：
1. **运行环境**：通过沙盒、虚拟机、文件系统边界、出口控制限制代理能做什么
2. **模型层**：系统提示、分类器、探测、训练修改
3. **外部内容**：MCP服务器、第三方插件、网络搜索工具

**防御哲学**：
> "When environmental defenses aren't available, the model layer has to pick up the slack"

### 1.2 三种隔离模式

#### 模式1：临时容器（claude.ai代码执行）
**架构特点**：
- 使用gVisor容器在隔离基础设施上运行
- 完全服务器端，无本地代码执行
- 文件系统是临时的（每次会话）
- 防御重点是保护基础设施和租户间隔离

**安全教训**：
> "the weakest layer is the one you built yourself"
> 最薄弱的层是自己构建的层

**漏洞发现**：
- 自定义代理是最容易出问题的部分
- 即使gVisor和seccomp已经过强化，自定义代码仍有未知风险

#### 模式2：人机循环沙盒（Claude Code）
**架构特点**：
- 在用户机器上运行，访问文件系统、shell和网络
- 早期设计：允许读取，写入、bash和网络访问需要批准
- 沙盒层（Seatbelt/bubblewrap）：允许读取和写入，默认拒绝网络

**风险遗漏**：
1. **信任对话框前的一切**：
   - 项目设置在显示"信任此文件夹？"提示前就解析
   - .claude/settings.json中的钩子在用户同意前就执行
   - **解决方案**：延迟本地配置解析和执行直到用户接受信任提示

2. **用户作为注入向量**：
   - 攻击者通过钓鱼邮件让员工执行恶意提示
   - 模型层防御锚定用户意图，无法检测此类攻击
   - **解决方案**：环境层防御（出口控制和文件系统边界）

#### 模式3：本地虚拟机（Claude Cowork）
**架构特点**：
- 在用户桌面上运行，访问用户选择的工作空间文件夹
- 使用完整虚拟机（Apple Virtualization/HCS）
- 独立的Linux内核、文件系统和进程表
- 用户凭据保留在主机keychain中

**架构改进**：
1. **主机模式**：将代理循环移出VM，保持代码执行在VM内
   - 解决VM启动失败导致服务不可用的问题
   - 提供重要安全保证：隔离代码执行

2. **本地MCP服务器**：移出VM，与Claude Desktop保持一致
   - 更易于审计
   - 避免VM更新时的脆弱依赖问题
   - 支持需要与本地进程交互的MCP

**文件系统控制**：
- 只读、读写、读写无删除三种挂载模式
- 符号链接解析必须在路径验证之前进行
- 企业客户可通过MDM设置挂载路径允许列表

**风险遗漏**：
- 通过批准域的渗出：恶意文件放置在用户挂载的工作空间中

### 1.3 核心安全原则

**分层防御**：
- 环境层防御失效时，模型层需要弥补
- 代理层和模型层可以共同防御恶意工具输出
- 工具限制可进一步缩小攻击范围

**安全教训总结**：
1. **本地不等于安全**：来自本地目录的输入应被视为来自互联网的请求
2. **用户是注入向量**：即使通过用户传递的恶意指令也需环境防御
3. **符号链接是攻击路径**：必须正确处理符号链接解析
4. **批准疲劳问题**：过多的批准提示导致用户注意力下降

## 二、Scaling Managed Agents: Decoupling the brain from the hands（2026-04-08）

**核心主题**：托管代理架构革新 - 将"大脑"（Claude及其Harness）与"手"（执行环境）解耦

### 2.1 问题背景

**早期问题**：
1. **宠物服务器困境**：所有组件放在单个容器中，服务器成为"宠物"
   - 容器失败时会话丢失
   - 容器无响应时需要手动恢复
   - 调试困难：无法区分Harness错误、事件流丢包或容器离线

2. **假设固化**：Harness假设所有资源都在容器内
   - 客户需要连接到VPC时，只能对等网络
   - 部署假设限制了灵活性

3. **性能问题**：每个会话都需要完整容器设置
   - 时间到首个标记（TTFT）过高
   - 不需要容器的会话也要支付启动成本

### 2.2 解耦架构设计

**核心概念**：
- **大脑（Brain）**：Claude及其Harness，负责推理和决策
- **手（Hands）**：沙盒和工具，负责执行具体操作
- **会话（Session）**：事件日志，记录所有发生的事情

**解耦优势**：
1. **故障隔离**：组件可以独立失败和替换
2. **灵活性**：大脑可以连接多个不同的手
3. **性能**：只在需要时才分配资源
4. **可扩展性**：可以轻松扩展多个大脑

### 2.3 安全架构革新

**凭证处理模式**：
- **与资源绑定**：将访问令牌与资源绑定（如Git）
- **外部保险库**：OAuth令牌存储在安全保险库中
- **代理模式**：通过专用代理访问外部服务，Harness永远不会接触凭据

**安全边界**：
- 确保令牌永远不会从Claude生成代码运行的沙盒中到达
- 使用MCP和令牌代理模式
- 避免令牌泄露攻击

### 2.4 会话设计哲学

**会话≠上下文窗口**：
- 长期任务超出上下文窗口长度
- 会话提供上下文对象，存在于上下文窗口之外
- `getEvents()`接口允许大脑查询事件流

**灵活上下文管理**：
- 可以选择性地获取上下文片段
- 可以在传递给Claude之前进行变换
- 分离了可恢复上下文存储和任意上下文管理

### 2.5 性能改进结果

**TTFT改进**：
- p50 TTFT下降约60%
- p95 TTFT下降超过90%
- 不需要容器的会话立即开始推理

**可扩展性**：
- 多个大脑意味着多个无状态Harness
- 只在需要时连接到手
- 容器通过工具调用按需分配

### 2.6 Meta-Harness设计理念

**核心原则**：
> "设计一个系统，适应未来的Harness、沙箱或其他组件"

**通用接口**：
- `execute(name, input) → string`：手的基本接口
- `wake(sessionId)`：重启Harness
- `getSession(id)`：获取事件日志
- `emitEvent(id, event)`：记录事件

**设计哲学**：
- 对接口有意见，对实现无意见
- 期待Claude需要操作状态和执行计算的能力
- 不对未来大脑或手的数量和位置做假设

### 2.7 对Harness Engineering的启示

**新的架构模式**：
1. **组件解耦**：大脑、手、会话相互独立
2. **接口抽象**：通用接口允许不同实现
3. **按需分配**：资源只在需要时分配
4. **安全边界**：凭据与执行环境隔离

**工程原则**：
1. **宠物vs牛群**：避免单点故障，采用可替换的组件
2. **上下文管理**：将上下文存储与上下文处理分离
3. **安全优先**：Harness不应该接触敏感凭据
4. **性能与安全平衡**：解耦既提升性能又增强安全性

## 四、实践指导

### 安全设计原则
1. **分层防御**：不要依赖单一安全层，使用多层次防御
2. **默认拒绝**：默认拒绝所有操作，显式允许必要操作
3. **隔离敏感数据**：凭据与执行环境隔离
4. **符号链接处理**：正确处理符号链接解析，防止路径遍历

### 架构设计原则
1. **组件解耦**：将大脑、手、会话作为独立组件
2. **接口抽象**：设计通用接口，允许实现变化
3. **按需分配**：资源只在需要时分配
4. **故障隔离**：组件独立失败和恢复

### 用户交互设计
1. **减少批准疲劳**：减少不必要的批准提示
2. **上下文感知**：在不同用户层级提供不同交互方式
3. **透明度**：让用户了解代理的访问范围
4. **默认安全**：默认采用最安全的配置
